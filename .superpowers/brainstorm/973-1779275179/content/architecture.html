3 cách tổ chức backend cho DevSense Web

A

Tất cả trên Colab/Local (1 backend)

Colab chạy 1 FastAPI duy nhất: auth, DB, crawl, inference. Frontend Railway gọi vào ngrok URL. Đơn giản nhất nhưng bất ổn — Colab restart mất DB, URL thay đổi.

Ưu

  • Code ít nhất
  • Deploy 1 chỗ

Nhược

  • URL ngrok thay đổi
  • Colab reset mất dữ liệu
  • Không scale
B

⭐ 3-tier tách biệt (Recommended)

Railway Main API: auth, user DB, device DB, favorites, proxy requests.
Colab/Local Worker: chỉ crawl + ABSA inference, gọi qua ngrok.
Railway Frontend: React app, luôn gọi Main API với URL cố định.

Frontend (Railway)
↓ fixed URL
Main API (Railway) — auth, DB, favorites
↓ proxy → ngrok URL (stored in DB)
Crawl Worker (Colab/Local) — scrape + ABSA

Ưu

  • URL frontend luôn cố định
  • DB an toàn trên Railway
  • Colab chỉ cần chạy khi crawl

Nhược

  • 2 backend cần maintain
  • Colab phải online khi crawl
C

Railway API + HuggingFace Inference

Railway Main API: auth, DB, crawl (Selenium local proxy), favorites.
HuggingFace Space: chạy ABSA model (như hiện tại với LLaMA).
Không cần Colab — nhưng crawl phải chạy local hoặc dùng proxy service.

Ưu

  • Không cần mở Colab
  • Inference luôn sẵn (HF Space)

Nhược

  • Crawl Shopee/Tiki từ Railway dễ bị block
  • HF Space free tier chậm/sleep
